# Recipes & Patterns

How-to reference for complete component builds. Each recipe combines 3–5 APIs from other pages into a realistic, production-shaped component. No new APIs are introduced here.

<Callout variant="warn" title="Static values only">
  Every style value throughout these examples must be a static string literal.
  Dynamic values — template literals, variables, ternaries, computed expressions
  — are skipped per-property at extraction time. The constraint applies equally
  inside `.styles()`, `.variant()`, `.states()`, and `.compound()`. See [Static
  Extraction & CSS Output](/docs/advanced/extraction) for the full skip model.
</Callout>

### Extension: ButtonLink

`.extend()` on the terminal component starts a new builder chain inheriting all base styles, variants, and states. Use `.asElement('a')` to change the underlying element. The extended component does not re-declare the parent's CSS — only the delta is emitted.

```tsx
// ButtonLink inherits Button's full style surface.
// .asElement('a') changes the rendered HTML element.
export const ButtonLink = Button.extend()
  .styles({
    // Override cursor for anchor semantics
    cursor: 'pointer',
  })
  .asElement('a');
```

```tsx
// Same variant/state props surface as Button
<ButtonLink href="/dashboard" intent="primary" size="md">
  Go to dashboard
</ButtonLink>

// System prop one-off — mt goes to @layer system, wins the cascade
<ButtonLink href="/signup" intent="ghost" mt={24}>
  Create account
</ButtonLink>
```

    Header: CardHeader,
    Body: WideCardBody,

<InputRoot>

<Callout variant="info" title="State vs. selector for form validity">
  `_invalid` fires on `:invalid`, `[aria-invalid="true"]`, and `[data-invalid]`
  — native CSS pseudo-class coverage plus headless library conventions. The
  `isInvalid` state prop adds the same visual styles via a class toggle, useful
  when your validation logic runs before the field has been touched and you need
  to suppress native `:invalid` on empty required fields.
</Callout>

## Color Scheme Patterns

Four techniques for mapping color to components. Each approach has a different trade-off in terms of cascade flexibility and theme adaptability.

### 1. Opacity syntax — alpha-adjusted token refs

The `{colors.key/N}` syntax resolves to `color-mix(in srgb, <token> N%, transparent)` in the extracted CSS. No hardcoded hex values.

```tsx
const SelectionCard = ds
  .styles({
    bg: 'surface',
    border: 1,
    borderColor: 'border',
    borderRadius: '6px',
    p: 16,
    transition: 'border-color 120ms ease, background-color 120ms ease',
  })
  .states({
    selected: {
      bg: '{colors.scheme.500/8}',
      borderColor: 'scheme.500',
      // Hover while selected — deepen the fill
      _hover: { bg: '{colors.scheme.500/14}' },
    },
    hovered: {
      bg: '{colors.scheme.500/5}',
      borderColor: '{colors.border/80}',
    },
  })
  .asElement('div');
```

```tsx
// scheme.500 resolves to fire.500 in dark mode, warm.300 in light mode —
// the component is theme-adaptive without any variant logic
<SelectionCard selected>Active item</SelectionCard>
```

### 2. CSS variable rebinding — scheme remapping per variant

The Button recipe above demonstrates this pattern. Instead of hardcoding colors in each variant, define CSS custom properties in `.styles()` with sensible defaults, then rebind them per variant. The cascade computes the correct color at render time.

```tsx
// Intent-colored badge — scheme variable rebound to a specific palette per intent
const Badge = ds
  .styles({
    display: 'inline-flex',
    alignItems: 'center',
    gap: '4px',
    fontWeight: '500',
    fontSize: 11,
    letterSpacing: '0.06em',
    textTransform: 'uppercase',
    borderRadius: '4px',
    px: 8,
    py: 4,
    // Defaults — overridden by each variant
    '--badge-bg': 'var(--colors-scheme-500)',
    '--badge-color': 'var(--colors-scheme-50)',
    backgroundColor: 'var(--badge-bg)',
    color: 'var(--badge-color)',
  })
  .variant({
    prop: 'intent',
    defaultVariant: 'default',
    variants: {
      default: {
        '--badge-bg': '{colors.border/40}',
        '--badge-color': 'var(--colors-text)',
      },
      primary: {
        '--badge-bg': '{colors.scheme.500/15}',
        '--badge-color': 'var(--colors-scheme-500)',
      },
      success: {
        '--badge-bg': '{colors.forest.500/12}',
        '--badge-color': 'var(--colors-forest-500)',
      },
      warning: {
        '--badge-bg': '{colors.gold.300/20}',
        '--badge-color': 'var(--colors-gold-300)',
      },
      danger: {
        '--badge-bg': '{colors.fire.500/12}',
        '--badge-color': 'var(--colors-fire-500)',
      },
      ocean: {
        '--badge-bg': '{colors.ocean.500/12}',
        '--badge-color': 'var(--colors-ocean-500)',
      },
    },
  })
  .asElement('span');
```

### 3. Contextual variable — current-bg

`current-bg` is a contextual variable declared on the theme via `declareContextualVars`. The `bg` prop sets it as a side effect — any child can read `var(--current-bg)` to match the parent's background.

```tsx
// Parent sets --current-bg as a side effect of bg prop
const Section = ds
  .styles({
    bg: 'bg',
    p: 48,
  })
  .asElement('section');

// Divider reads the parent's background to create a cut-out effect
const SectionDivider = ds
  .styles({
    height: '1px',
    backgroundColor: 'border',
    // --current-bg set by the nearest parent with bg prop
    boxShadow: '0 0 0 4px var(--current-bg)',
  })
  .asElement('hr');

// Inset ring that matches parent background — no hardcoded color
const InsetRing = ds
  .styles({
    outline: '2px solid',
    outlineColor: 'border',
    outlineOffset: '-4px',
    // Inset ring uses parent bg via contextual var
    boxShadow: '0 0 0 8px var(--current-bg)',
  })
  .asElement('div');
```

### 4. Comparison — same visual, three approaches

The visual: a chip component with a subtle tinted background matching the color mode's scheme.

```tsx
// Approach A: direct opacity token ref in .styles()
// Trade-off: single visual, not switchable without a variant
const ChipA = ds
  .styles({
    bg: '{colors.scheme.500/8}',
    color: 'scheme.500',
    px: 8,
    py: 4,
    borderRadius: '999px',
    fontSize: 12,
    fontWeight: '500',
  })
  .asElement('span');

// Approach B: variant for multiple palettes
// Trade-off: explicit control, more declarations
const ChipB = ds
  .styles({
    px: 8,
    py: 4,
    borderRadius: '999px',
    fontSize: 12,
    fontWeight: '500',
  })
  .variant({
    prop: 'palette',
    defaultVariant: 'scheme',
    variants: {
      scheme: { bg: '{colors.scheme.500/8}', color: 'scheme.500' },
      fire: { bg: '{colors.fire.500/8}', color: 'fire.500' },
      ocean: { bg: '{colors.ocean.500/8}', color: 'ocean.500' },
      forest: { bg: '{colors.forest.500/8}', color: 'forest.500' },
    },
  })
  .asElement('span');

// Approach C: state prop for a single on/off tint toggle
// Trade-off: additive — layers over existing base color
const ChipC = ds
  .styles({
    display: 'inline-flex',
    px: 8,
    py: 4,
    borderRadius: '999px',
    fontSize: 12,
    fontWeight: '500',
    color: 'text.muted',
    bg: '{colors.border/30}',
  })
  .states({
    active: {
      bg: '{colors.scheme.500/12}',
      color: 'scheme.500',
    },
  })
  .asElement('span');
```

**When to reach for each:**

| Approach                           | Use when                                                                     |
| ---------------------------------- | ---------------------------------------------------------------------------- |
| Direct opacity ref in `.styles()`  | One visual, always the same tint, theme-adaptive                             |
| CSS variable rebinding per variant | Multiple palette options, interaction hover states share declarations        |
| State prop                         | Toggle between neutral and tinted — binary, additive, layered on top of base |
| `current-bg` contextual var        | Child needs to "punch through" to parent background without coordination     |
