# Theming & Tokens

The theme builder converts nested token definitions into a serialized contract that the Rust extraction crate reads at build time. Each token becomes a CSS custom property. The chain enforces definition order at the type level — you cannot call `addScale` before `addBreakpoints` because the type state won't permit it.

## The Theme Builder Chain

`createTheme()` returns a fresh `ThemeBuilder`. The chain is a sequence of steps, each returning a new builder instance with an accumulated type state. Call `build()` to finalize.

<ChainStep steps={[
  {
    label: 'createTheme()',
    layer: 'theme',
    description: 'Returns a ThemeBuilder with an empty type state. Entry point — no configuration yet.',
    code: `import { createTheme } from '@animus-ui/system';\n\nconst tokens = createTheme()`,
  },
  {
    label: 'addBreakpoints()',
    layer: 'breakpoints',
    description: 'Registers breakpoint keys and their pixel values. Must be called first — the type state enforces this.',
    code: `const tokens = createTheme()\n  .addBreakpoints({ sm: 768, lg: 1280 })`,
  },
  {
    label: 'addColors()',
    layer: 'colors',
    description: 'Registers color palette entries. Always emits CSS variables — every color becomes --color-*.',
    code: `  .addColors({\n    ember: '#FF2800',\n    gray: { 50: '#fafafa', 900: '#111' },\n  })`,
  },
  {
    label: 'addScale()',
    layer: 'scale',
    description: 'Registers a named scale. Emits CSS variables only when emit: true is set.',
    repeatable: true,
    code: `  .addScale({ name: 'space', values: { 4: '0.25rem', 8: '0.5rem', 16: '1rem' }, emit: true })\n  .addScale({ name: 'fontSizes', values: { sm: '0.875rem', base: '1rem', lg: '1.25rem' } })`,
  },
  {
    label: 'build()',
    layer: 'finalize',
    description: 'Flattens the accumulated state. Attaches manifest, serialize(), and varRef() as non-enumerable properties on the returned object.',
    code: `  .build();\n\nexport { tokens };`,
  },
]} />

<Callout variant="info" title="Type constraint">
  `addBreakpoints` is not just a convention — the return type of `createTheme()` only exposes `addBreakpoints`. Other methods are not present until breakpoints are registered. This prevents you from omitting them.
</Callout>

## Scale Definition & Emission

Not all scales produce CSS variables. The distinction is tracked by the `Emitted` generic parameter, which accumulates through the chain and powers type-level token ref validation.

| Call | Emits CSS variables? |
|------|---------------------|
| `addColors({ ... })` | Always — every key becomes `--color-*` |
| `addScale({ ..., emit: true })` | Yes — every key becomes `--{name}-*` |
| `addScale({ ... })` | No — raw values only, no variables |

When a scale is not emitted, its tokens resolve to their raw values at extraction time. When emitted, tokens resolve to `var(--{scale}-{key})` — the CSS variable reference is embedded in the generated class, not the value itself.

<CodeExample
  input={`createTheme()
  .addBreakpoints({ sm: 768, lg: 1280 })
  .addColors({
    ember: '#FF2800',
    gray: { 50: '#fafafa', 900: '#111827' },
  })
  .addScale({
    name: 'space',
    values: { 4: '0.25rem', 8: '0.5rem', 16: '1rem' },
    emit: true,
  })
  .addScale({
    name: 'fontSizes',
    values: { sm: '0.875rem', base: '1rem' },
    // emit omitted — raw values only
  })
  .build()`}
  output={`:root {
  --color-ember: #FF2800;
  --color-gray-50: #fafafa;
  --color-gray-900: #111827;
  --space-4: 0.25rem;
  --space-8: 0.5rem;
  --space-16: 1rem;
  --breakpoint-sm: 768px;
  --breakpoint-lg: 1280px;
}

/* fontSizes: no CSS variables emitted */
/* fontSizes.sm → "0.875rem" in scalesJson only */`}
  inputLabel="theme definition"
  outputLabel="emitted CSS"
/>

### Phantom Scale Types

When a scale consists of open-ended values — numeric IDs or arbitrary strings — use the helper factories to constrain what the type system accepts for `values`:

<TypeSignature
  name="createScale"
  generics="T extends string | number"
  params={[]}
  returns="readonly T[]"
/>

`createScale<T>()` is a phantom type factory. It returns an empty array at runtime and is discarded after the type is inferred. Use it to declare a scale that accepts a specific literal union.

```ts
import { createScale } from '@animus-ui/system';

// Constrains the scale to accept only these exact keys
const radiusScale = createScale<'none' | 'sm' | 'full'>();
```

The three pre-built open scale variants accept any value of their respective type:

| Export | Accepts |
|--------|---------|
| <TokenBadge variant="method">numericScale</TokenBadge> | `number & {}` — any number |
| <TokenBadge variant="method">stringScale</TokenBadge> | `string & {}` — any string |
| <TokenBadge variant="method">numericOrStringScale</TokenBadge> | `(string \| number) & {}` — either |

```ts
import { numericScale, stringScale } from '@animus-ui/system';

createTheme()
  .addBreakpoints({ sm: 768 })
  .addScale({ name: 'zIndex', values: numericScale, emit: false })
  .addScale({ name: 'easing', values: stringScale, emit: false })
```

## Token Path Syntax

Tokens appear in two syntactic forms depending on context.

### Bare keys — typed props

When a prop has a known scale binding (e.g., `fontSize` → `fontSizes`), pass the bare key:

```tsx
ds.styles({
  fontSize: 'sm',       // ✓ bare key — typed against fontSizes scale
  color: 'ember',       // ✓ bare key — typed against colors scale
  space: 4,             // ✓ numeric bare key — typed against space scale
})
```

The system resolves `'sm'` to `var(--fontSize-sm)` (if emitted) or `'0.875rem'` (if raw) at extraction time.

### Template syntax — untyped contexts

When assigning to an arbitrary CSS property or custom property, use the `{scale.key}` template:

```tsx
ds.styles({
  boxShadow: '0 0 12px {colors.ember/40}',   // ✓ template in untyped context
  '--current-surface': '{colors.gray.900}',   // ✓ template for custom property
})
```

<Callout variant="info" title="Opacity modifier">
  Append `/{alpha}` to any color token ref to apply transparency via `color-mix`. `{colors.ember/40}` resolves to `color-mix(in srgb, var(--color-ember) 40%, transparent)`. Alpha of `0` collapses to `transparent`; `100` passes through unchanged.
</Callout>

Both forms resolve to the same CSS variable when the scale is emitted. The distinction is purely about what the type system knows at the call site.

### ColorTokenRef and ScaleTokenRef

<APIBlock>
  <ParamTable params={[
    { name: 'ColorTokenRef', type: 'template literal', default: '—', desc: 'Template literal type for {colors.key} and {colors.key/alpha} patterns. Computed from the augmented Theme interface.' },
    { name: 'ScaleTokenRef<E>', type: 'template literal', default: '—', desc: 'Constrains {scale.key} where scale is limited to the emitted scale names E.' },
  ]} />
</APIBlock>

`ColorTokenRef` is a computed type. When you augment the `Theme` interface, `ColorTokenRef` narrows to only the keys present in your `colors` scale:

```ts
// ColorTokenRef with augmented theme:
// '{colors.ember}' | '{colors.ember/number}' | '{colors.gray.50}' | ...

const ref: ColorTokenRef = '{colors.ember/80}'; // ✓
const bad: ColorTokenRef = '{colors.unknown}';   // ✗ — type error
```

`ScaleTokenRef<E>` is a looser pattern — it permits any dot-path under the given scale union without enumerating keys. Used internally by style props that accept cross-scale references.

## Serialization Contract

Calling `.build()` finalizes the theme and returns a `BuiltTheme`. Three non-enumerable properties are attached via `Object.defineProperty` — they do not appear in iteration but are accessible by name:

| Property | Description |
|----------|-------------|
| `manifest` | Structured in-memory manifest. Consumed by the Vite plugin. |
| `serialize()` | Returns `SerializedTheme` — JSON strings for pipeline consumption. |
| `varRef(tokenPath)` | Resolves a dot-path token to its `var()` reference string. |

<Callout variant="warn" title="Non-enumerable">
  `manifest`, `serialize`, and `varRef` are attached as non-enumerable, non-configurable, non-writable properties on the built theme object. They do not appear in `Object.keys()`, `JSON.stringify()`, or spread operations. Access them directly by name.
</Callout>

### SerializedTheme

`tokens.serialize()` produces the four fields consumed by the extraction pipeline:

<APIBlock>
  <ParamTable params={[
    { name: 'scalesJson', type: 'string (JSON)', default: '—', desc: 'Flattened token map. "space.8" → "0.5rem", "breakpoints.sm" → "768". Consumed by Rust theme_resolver to substitute bare scale key references.' },
    { name: 'variableMapJson', type: 'string (JSON)', default: '—', desc: 'Token path → CSS variable name without var() wrapper. "colors.ember" → "--color-ember". Consumed by Rust for {scale.key} token alias resolution.' },
    { name: 'variableCss', type: 'string (CSS)', default: '—', desc: 'Pre-built :root { --color-*: ... } block plus [data-color-mode] override blocks for each registered mode. Injected into virtual:animus/styles.css by the Vite plugin.' },
    { name: 'contextualVarsJson', type: 'string (JSON)', default: '—', desc: 'Per-scale registry of contextual variable names. Consumed by Rust contextual resolution for runtime-resolved CSS variables.' },
  ]} />
</APIBlock>

The Vite plugin calls `tokens.serialize()` once during `buildStart`. The Rust crate receives `scalesJson` and `variableMapJson` as inputs to its resolver. `variableCss` is injected verbatim into the virtual CSS module.

### manifest

`tokens.manifest` is the unserialized form of the same data:

<APIBlock>
  <ParamTable params={[
    { name: 'tokenMap', type: 'Record<string, string>', default: '—', desc: 'Flat dot-path → resolved value. Emitted scales carry var() references; raw scales carry raw values.' },
    { name: 'variableMap', type: 'Record<string, string>', default: '—', desc: 'Dot-path → CSS variable name (without var() wrapper) for emitted tokens only.' },
    { name: 'modes', type: 'Record<string, Record<string, string>>', default: '—', desc: 'Mode name → flat token map for that mode\'s alias overrides. Populated by addColorModes().' },
    { name: 'variableCss', type: 'string', default: '—', desc: ':root and [data-color-mode] CSS blocks, pre-built.' },
    { name: 'contextualVars', type: 'Record<string, string[]> | undefined', default: 'undefined', desc: 'Scale name → array of contextual var names. Present only when declareContextualVars() was called.' },
  ]} />
</APIBlock>

### varRef

`tokens.varRef(tokenPath)` resolves a dot-path token at runtime, useful for reading theme values in application code:

```ts
const built = createTheme()
  .addBreakpoints({ sm: 768 })
  .addColors({ ember: '#FF2800' })
  .addScale({ name: 'space', values: { 8: '0.5rem' }, emit: true })
  .build();

built.varRef('colors.ember');  // → 'var(--color-ember)'
built.varRef('space.8');       // → 'var(--space-8)'
built.varRef('fontSizes.sm');  // → '0.875rem' (raw — not emitted)
built.varRef('colors.missing'); // → undefined
```

For emitted scales, `varRef` returns the `var()` reference string. For non-emitted scales, it returns the raw string value. Returns `undefined` if the path does not exist.

## Contextual Variables

Contextual variables solve a specific problem: style values that must be set dynamically at runtime (e.g., a component's current background color as a CSS variable for a child to read) but whose names the extraction crate needs to know in advance.

<MethodCard
  name="declareContextualVars"
  description="Registers per-scale slot names as phantom type extensions. These names are available as typed keys in style props but are not emitted as theme CSS variables."
  returnType="ThemeBuilder"
  available="addColors() or addScale()"
  example={
    <CodeExample
      input={`createTheme()
  .addBreakpoints({ sm: 768 })
  .addColors({
    ember: '#FF2800',
    gray: { 50: '#fafafa', 900: '#111827' },
  })
  .declareContextualVars({
    colors: ['bg-current', 'fg-current'],
  })
  .build()`}
      output={`/* No new :root variables emitted */
/* The extraction crate knows:
   colors.bg-current → --color-bg-current (available)
   colors.fg-current → --color-fg-current (available) */

/* Consumer sets them at component scope: */
.my-component {
  --color-bg-current: var(--color-ember);
}`}
      inputLabel="theme definition"
      outputLabel="runtime pattern"
    />
  }
/>

<ParamTable params={[
  { name: 'vars', type: 'Partial<{ [K in keyof T]: readonly string[] }>', default: 'required', desc: 'Map of scale name to array of contextual variable names. Scale must already exist (addColors or addScale must have been called for that scale).' },
]} />

`declareContextualVars` performs a phantom type merge — the declared names are added to the scale type as `Record<name, \`var(--${string})\`>` so they appear as valid typed keys in `.styles()` calls. No runtime data is added to the theme object.

The names are serialized in `contextualVarsJson` and consumed by the Rust crate to allow `{colors.bg-current}` template references in extracted components, where the actual variable value is set by the consuming component at render time.

<Callout variant="info">
  The scale must exist before you call `declareContextualVars` — passing an undeclared scale name throws at build time. The constraint is enforced at the type level: `Vars` is typed as `Partial<{ [K in keyof T]: ... }>`, so TypeScript rejects unknown scale names.
</Callout>
