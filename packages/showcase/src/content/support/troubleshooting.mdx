# Troubleshooting

---

## Debugging Decision Tree

Start here. Match your symptom, run the fix.

| Symptom | Cause | Fix |
|---------|-------|-----|
| Styles not updating in dev | `buildStart` results are held in memory — plugin source changes do not hot-reload | Restart the dev server |
| Transforms seem stale | Stale Vite transform cache at `node_modules/.vite/` | `bun run clean:light` |
| NAPI function errors / wrong arity | Compiled `.node` binary is out of sync with current Rust source | `bun run rebuild` |
| Showcase builds but styles are missing | `virtual:animus/styles.css` not loaded, or the import was tree-shaken | Open browser devtools → Network tab → filter `styles.css` — verify the virtual module loaded and inspect its content |
| CSS has `__TRANSFORM__` strings | Transform resolution subprocess failed after NAPI analysis | Check terminal for `[animus-extract]` warnings; ensure all transforms are registered by name |
| Nothing works | Unknown corruption — cache, binary, or build artifact mismatch | `bun run rebuild` (clean:full + build:all from scratch) |

---

## Cache Clearing

Three cache tiers with different cost/scope tradeoffs.

| Command | Removes | Speed | When to use |
|---------|---------|-------|-------------|
| `bun run clean:light` | `.vite` cache + `dist/` | Fast (under 1s) | Transforms seem stale, styles not updating after clean restarts |
| `bun run clean:full` | Above + Rust `target/` + `.node` binary | Slow (rebuild 30–60s) | NAPI errors, arity mismatches, binary signature changes |
| `bun run clean` | `dist/` + `target/` only | Fast | Legacy alias — prefer `clean:light` or `clean:full` |

<Callout variant="info" title="When to use clean:full vs rebuild">
  `bun run clean:full` removes all build artifacts. `bun run rebuild` goes further: it runs `clean:full` then immediately rebuilds the full pipeline (`build:all`). Use `rebuild` when you need a verified working state, not just a clean slate.
</Callout>

## Silent Failure Diagnostics

These conditions produce incorrect output with no console errors. They require active inspection.

### Scale miss passthrough

When a style value references a token key that does not exist in the theme, the Rust crate passes the raw string through as a literal CSS value. No warning is emitted.

```tsx
// "nonExistentToken" is not in the colors scale
ds.styles({ bg: 'nonExistentToken' })
```

```css
/* Output — browser ignores this, no error in console */
.animus-MyComponent-a1b2 {
  background: nonExistentToken;
}
```

**Fix:** Audit the theme scale for the property. Use `verbose: true` during development to surface skip diagnostics.

### Unknown selector aliases

Selector aliases (e.g., `_hover`, `_groupHover`) that are not registered in the design system are silently dropped. The prop is not included in the extracted CSS.

```tsx
// _groupHover is only valid if registered on the system instance
ds.styles({ _groupHover: { color: 'primary' } })
```

**Fix:** Verify the alias exists in your `ds.ts` system definition. Check that `addSelectors()` or equivalent was called with the alias name.

### JSX spread hides variant props from the scanner

`<Button {...props} />` cannot be statically analyzed. The JSX scanner cannot see which variant and state props are passed through a spread. The reconciler records zero usage for any variant option applied exclusively via spread, and prunes it before CSS emit.

```tsx
// Variant "danger" never appears as a literal prop in JSX
// → pruned → no CSS generated for intent="danger"
<Button {...buttonProps} />

// This records usage — "danger" will extract
<Button intent="danger" />
```

**Fix:** Apply variant props as direct literal attributes wherever possible. If a variant must be spread, accept that its CSS must be handled at runtime.

### Per-property skip — partial extraction

When a style object contains a non-static value (identifier, ternary, function call, template literal), the evaluator skips that property and continues. The component still extracts partially — static properties in the same object are not affected.

```tsx
ds.styles({
  color: 'text',         // extracted — static string
  fontSize: 14,          // extracted — static number
  borderColor: themeVar, // skipped — non-static identifier
  p: 16,                 // extracted — continues after skip
})
```

Enable `verbose: true` (or `ANIMUS_DEBUG=1`) to see `[skip]` markers in extraction output identifying which properties were not extracted.

<Callout variant="warn" title="Partial skip is per-property, not per-component">
  A single non-static value does not bail the entire component. Only that property is excluded from the static CSS layer. If you expect a component to produce no CSS, inspect verbose output — static properties in the same object will still extract.
</Callout>

## Common Error Messages

### NAPI function errors / wrong arity

**Cause:** The compiled `.node` binary does not match the current Rust source. This happens after pulling changes that update the Rust crate without recompiling.

**Fix:**
```bash
bun run rebuild
```

### `__TRANSFORM__` strings in compiled CSS

**Cause:** The transform resolution subprocess failed after NAPI analysis. The crate emits `__TRANSFORM__{name}__{rawValue}__` markers as placeholders; a bun subprocess is responsible for substituting them with the result of the named transform function. If that subprocess errors, the placeholders survive into the output stylesheet.

**Fix:** Check the terminal for `[animus-extract]` prefixed warnings. Common root causes:
- The system file (`ds.ts`) could not be loaded by the subprocess
- A named transform function threw during resolution
- An anonymous (inline) transform was used — these cannot be retrieved by name and fall back to the raw value

Ensure all transforms used in extraction are registered by name via the system builder:

```typescript
// ds.ts — named, retrievable by the transform subprocess
createSystem().addTransforms({ fluid: (v) => `clamp(...)` })

// anonymous — NOT retrievable, placeholder survives as raw value
createSystem().addGroup({ size: { transform: (v) => `clamp(...)` } })
```

### `[skip]` markers in verbose output

**Cause:** The style evaluator encountered a non-static property value. This is expected degradation, not an error.

**Fix:** If the skipped property needs to extract statically, replace the dynamic value with a static string literal or a token reference. See [Per-Property Skip](#per-property-skip--partial-extraction) above.

### Component eliminated warning

**Symptom:** Terminal shows `⚠ ComponentName eliminated` during build.

**Cause:** The reconciler found that a component extracted zero usable styles after JSX scanning. The component is pruned from the manifest.

**Common triggers:**
- Component is defined but never rendered anywhere in the source tree
- All variant options are only applied via JSX spreads (see [JSX spread](#jsx-spread-hides-variant-props-from-the-scanner))
- Component uses `animus.styles()` from `@animus-ui/core` instead of `ds.styles()` from the system instance

**Fix:** Verify the component is rendered with literal props. Confirm it uses the system instance (`ds`) not the bare `animus` builder.

### Subprocess spawn error

**Cause:** `bun` is not installed or not on `PATH` at build time. The plugin spawns bun subprocesses for system loading and transform resolution.

**Fix:** Verify `bun` is on PATH:
```bash
which bun
bun --version
```

If running in CI, ensure bun is installed in the CI environment before the build step.

## Reading Extracted Output

### Inspect the virtual stylesheet

Open browser devtools and filter the Network tab for `styles.css`. The virtual module `virtual:animus/styles.css` loads as a single concatenated stylesheet with three sections:

1. **Variable CSS** — `:root { --color-*: ... }` plus `[data-color-mode]` selectors (unlayered, highest precedence)
2. **Global CSS** — `@layer global { }` — reset and global blocks from `.withGlobalStyles()`
3. **Component CSS** — `@layer base, variants, compounds, states, system, custom { }`

If the network request is absent, `virtual:animus/styles.css` was not imported in your app entry point. Add:

```typescript
// main.tsx
import 'virtual:animus/styles.css';
```

### Class name anatomy

```
animus-Button-a1b2c3
       │      └── hash of "src/components/Button.tsx::Button"
       └────────── binding name (component variable name)
```

The hash is derived from the file path and binding name — not from style values. Renaming a component or moving it to a different file changes the hash. Changing a style value does not.

### State and variant classes

State classes use a double-dash separator between the component name and the state name:

```
animus-Button-a1b2c3--disabled
animus-Button-a1b2c3--loading
```

Variant option classes include both the prop name and the value:

```
animus-Button-a1b2c3--intent-primary
animus-Button-a1b2c3--intent-danger
animus-Button-a1b2c3--size-sm
```

If an expected state or variant class is absent from the rendered element's class list, the component definition or JSX usage may have an extraction problem. Check the virtual stylesheet for the class definition — if it is absent from the CSS entirely, the option was pruned (see [JSX spread](#jsx-spread-hides-variant-props-from-the-scanner)).

<Callout variant="info" title="Enable verbose logging">
  `ANIMUS_DEBUG=1 bun run dev` (or `verbose: true` in the plugin config) enables phase checkpoint output, per-file transform logs, file discovery counts, and HMR decision traces. All output is prefixed with `[animus]` for grep filtering. This is the primary diagnostic tool for extraction issues.
</Callout>
