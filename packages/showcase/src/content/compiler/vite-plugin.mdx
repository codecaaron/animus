# Vite Plugin

`animusExtract` bridges the Rust extraction crate with your Vite build. It runs in both dev and production: loads your design system via a bun subprocess, runs project-level static analysis, serves extracted CSS via a virtual module, and transforms each source file to replace builder chains with `createComponent()` calls.

## Installation & Basic Config

```bash
bun add -D @animus-ui/vite-plugin
```

```typescript
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import { animusExtract } from '@animus-ui/vite-plugin';

export default defineConfig({
  plugins: [react(), animusExtract({ system: './src/ds.ts' })],
});
```

<Callout variant="danger" title="Plugin order">
  `animusExtract` must come after `react()`. The React plugin transforms JSX
  first; Animus then walks the resulting AST to find builder chains and JSX
  component usage.
</Callout>

<Callout variant="danger" title="No resolve aliases">
  Never add `resolve.alias` entries for React or any package imported by your
  extracted components. Vite discards transform hook results for aliased
  modules, silently breaking the extraction pipeline.
</Callout>

<Callout variant="warn" title="bun required">
  The plugin loads your system module and resolves transforms via `bun`
  subprocesses. `bun` must be on `PATH` at build time. This is required for ESM
  isolation — the system module and its dependencies cannot be imported directly
  into Vite's plugin context.
</Callout>

### Layer Order

Animus emits CSS into 7 layers in a fixed cascade order. If you need to interleave app layers, pass a full `layers` array:

<APIBlock>
```typescript
// Without prefix — use Animus names directly
animusExtract({
  system: './src/ds.ts',
  layers: [
    'global',
    'base',
    'variants',
    'compounds',
    'states',
    'system',
    'custom',
    'overrides',   // your app layer, interleaved after Animus layers
  ],
});
```
```typescript
// With prefix — Animus names get the prefix, your layers do not
animusExtract({
  system: './src/ds.ts',
  prefix: 'acme',
  layers: [
    'acme-global',
    'acme-base',
    'acme-variants',
    'acme-compounds',
    'acme-states',
    'acme-system',
    'acme-custom',
    'overrides',
  ],
});
```
</APIBlock>

All 7 Animus layers must appear as a subsequence in their required order. Missing or reordered layers throw at build time.

## Virtual Modules

The plugin serves two virtual modules over Vite's module graph.

### `virtual:animus/styles.css`

The compiled stylesheet. Import it once in your app entry point.

```typescript
// main.tsx
import 'virtual:animus/styles.css';
```

The module concatenates three CSS blocks in order:

1. **Variable CSS** — `:root { --color-*: ... }` plus color mode selectors (`[data-color-mode="dark"] { ... }`)
2. **Global CSS** — `@layer global { }` containing your `.withGlobalStyles()` reset and global blocks
3. **Component CSS** — `@layer base / variants / compounds / states / system / custom { }` extracted from source files

All three blocks are post-processed by Lightning CSS: autoprefixing, syntax lowering, and optional minification.

### `virtual:animus/system-props`

Internal module. Serves the group prop map JSON consumed by the runtime bridge. Do not import this directly — it is automatically injected by the transform hook.

### Dev HMR

In dev mode, the plugin invalidates `virtual:animus/styles.css` whenever a transformed source file changes. The HMR bridge delivers CSS updates without a full page reload for component style changes.

**Content-hash skip:** Files that have not changed since the last analysis are passed to the Rust crate with an empty source string. The crate returns the cached result without reparsing. Only the changed file's source is transmitted across the NAPI boundary.

**Geological reset:** If the system file (`ds.ts`) changes, the plugin triggers a full reload — re-running the bun subprocess to reload the system, re-evaluating all tokens, and re-running project analysis from scratch.

### Subprocess Model

The plugin spawns bun subprocesses for ESM isolation. The system module uses top-level imports that cannot be loaded into Vite's CJS plugin context directly.

| Subprocess           | Purpose                                                          | Trigger                                   |
| -------------------- | ---------------------------------------------------------------- | ----------------------------------------- |
| System load          | Import system module, call `.serialize()`                        | `buildStart`, geological reset            |
| Global styles        | Resolve prop shorthand (bg → background-color) with transforms   | `buildStart`, after system load           |
| Transform resolution | Apply `__TRANSFORM__` placeholders with live transform functions | After analysis, if named transforms exist |

## Debugging & Diagnostics

### Enable Verbose Logging

```bash
# Environment variable — no config change needed
ANIMUS_DEBUG=1 bun run dev

# Or in config
animusExtract({ system: './src/ds.ts', verbose: true })
```

Verbose mode enables phase checkpoint output, per-file transform logs, file discovery counts, and HMR decision traces. All output is prefixed with `[animus]` for grep filtering.

**Always-on diagnostics** (no verbose needed): reconciliation elimination warnings (`⚠ ComponentName eliminated`) appear whenever the pipeline prunes a component that extracted zero usable styles.

### Diagnosing Common Failures

<Callout variant="warn" title="__TRANSFORM__ placeholders in CSS output">
  If the compiled stylesheet contains literal `__TRANSFORM__` strings, the
  transform resolution subprocess failed. Check the terminal for `[animus]`
  warnings. This typically means a named transform function threw an error or
  the system file could not be loaded.
</Callout>

<Callout variant="warn" title="[skip] markers in extraction errors">
  The style evaluator degrades gracefully: when a property value is non-static
  (identifier, function call, ternary), it skips that property and continues.
  `[skip]` markers in extraction output identify which properties were not
  extracted. The component still works at runtime via dynamic props — only the
  static CSS layer is missing.
</Callout>

<Callout variant="info" title="Stale .vite cache">
  The Vite dev cache at `node_modules/.vite/` stores pre-transformed module
  results across restarts. After changing NAPI function signatures or plugin
  behavior, the cache may serve stale transforms even though plugin logs look
  correct. Fix: `bun run clean:light` or `rm -rf node_modules/.vite/`.
</Callout>

### Diagnostic Decision Tree

| Symptom                              | Fix                                                                                               |
| ------------------------------------ | ------------------------------------------------------------------------------------------------- |
| Styles not updating in dev           | Restart the dev server                                                                            |
| Transforms seem stale                | `bun run clean:light`                                                                             |
| NAPI function errors / wrong arity   | `bun run rebuild`                                                                                 |
| CSS has `__TRANSFORM__` strings      | Transform subprocess failed — check terminal for `[animus]` warnings                              |
| Component renders but has no styles  | Check that it uses `ds.styles()` (system instance), not `animus.styles()` (core)                  |
| CI build passes but styles are wrong | Add `strict: true` to surface extraction failures as errors                                       |
| Styles missing after system changes  | Restart dev server; system file changes trigger geological reset but plugin source changes do not |
