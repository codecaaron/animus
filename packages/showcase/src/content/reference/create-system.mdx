# createSystem()

`createSystem()` returns a `SystemBuilder` — an immutable, type-state accumulator that registers props, groups, and selector aliases. Every method returns a new builder instance with widened generics. `.build()` finalizes the accumulator into a `SystemInstance` and a `createGlobalStyles` factory.

One system per application. Every component in the app imports from the built system instance.

```ts
import { createSystem } from '@animus-ui/system';

export const { system: ds, createGlobalStyles } = createSystem()
  .addGroup('surface', { ...color, ...border })
  .addGroup('space', space)
  .build();
```

  returnType="SystemBuilder<PropReg, GroupReg>"
/>

<APIBlock>

    params={[{ name: 'selectors', type: 'Record<string, string>' }]}
    returns="SystemBuilder<PropReg, GroupReg>"
  />
  <ParamTable
    params={[
      {
        name: 'selectors',
        type: 'Record<string, string>',
        default: 'required',
        desc: 'Map of alias key (including the leading underscore) to CSS selector string. Comma-separate compound selectors. The underscore is part of the key — it is stored and matched verbatim.',
      },
    ]}
  />
</APIBlock>

Each entry in the map is either an override or a new alias:

- **Override** — an alias key that matches a built-in name replaces the selector string but preserves the original cascade order position.
- **New alias** — an alias key not present in the built-ins is assigned cascade order starting at 500, incrementing by 10 per new entry.

Cascade order controls which state rule wins when two selectors apply at equal specificity. Built-in orders follow LVHA convention (link=10, hover=30, focus=50, active=70). Positional selectors (first, last) are at 400–410.

```ts
createSystem().addSelectors({
  // Override: replaces selector string, built-in cascade order (30) preserved
  _hover: '&:hover, &[data-hover]',
  // New aliases: order 500, 510, 520
  _groupHover: '.group:hover &',
  _dark: '[data-color-mode="dark"] &',
  _print: '@media print &',
});
```

<Callout variant="warn" title="Key format">
  Alias keys are stored verbatim and matched verbatim in `.states()` calls.
  `_hover` in `addSelectors` registers as `_hover` — the underscore is part of
  the key, not a separate convention.
</Callout>

### Collision detection

Group names and prop names share a flat namespace. The builder throws at runtime if:

- The group `name` is already registered as a prop name.
- Any prop in `config` is already registered as a group name.

```ts
createSystem()
  .addGroup('space', space)    // registers props: m, p, mx, px, ...
  .addGroup('space', { ... })  // throws — "space" already a group
  .addProps({ space: { ... } }) // throws — "space" already a group name
```

### Overlap tolerance

A prop name that already exists in the registry is allowed in `config` **if and only if the definition is identical** — same `property`, `scale`, `transform`, and `negative`. This lets multiple groups share a prop (e.g. `color` appears in both a `surface` group and a `text` group) without duplication.

The builder throws when a prop is re-registered with a different definition:

```ts
createSystem()
  .addGroup('surface', {
    color: { property: 'color', scale: 'colors' },
  })
  .addGroup('text', {
    // Allowed: identical definition
    color: { property: 'color', scale: 'colors' },
    // Would throw: different scale
    // color: { property: 'color', scale: 'textColors' },
  });
```

### Pre-built groups

Pre-built groups are available from `@animus-ui/system/groups`:

```ts
import {
  background,
  border,
  color,
  flex,
  grid,
  layout,
  positioning,
  shadows,
  space,
  transitions,
  typography,
} from '@animus-ui/system/groups';

createSystem()
  .addGroup('surface', { ...color, ...border, ...shadows })
  .addGroup('arrange', { ...flex, ...grid, ...layout })
  .addGroup('space', space)
  .build();
```

Spread pre-built group objects into `addGroup` config. Add inline props alongside the spread — append `as const` to entries with a `transform` key so TypeScript narrows the transform type:

```ts
.addGroup('layout', {
  ...flex,
  ratio: { property: 'aspectRatio', transform: ratio } as const,
})
```

## addProps()

<MethodCard
  name="addProps"
  description="Register individual props without a group — available on all components but not exposed as a group handle"
  returnType="SystemBuilder<PropReg & Conf, GroupReg>"
/>

<APIBlock>
  <TypeSignature
    name="addProps"
    generics="Conf extends Record<string, Prop>"
    params={[{ name: 'config', type: 'Conf' }]}
    returns="SystemBuilder<PropReg & Conf, GroupReg>"
  />
  <ParamTable
    params={[
      {
        name: 'config',
        type: 'Record<string, Prop>',
        default: 'required',
        desc: 'Props to register outside any group. Prop names must not collide with any registered group name. The same overlap tolerance rules as addGroup() apply — identical redefinitions are allowed, divergent ones throw.',
      },
    ]}
  />
</APIBlock>

`addProps` registers props that belong to no group. They appear on every component's system props surface but cannot be accessed via a group shorthand.

```ts
createSystem()
  .addProps({
    cursor: { property: 'cursor' },
    userSelect: { property: 'userSelect' },
    pointerEvents: { property: 'pointerEvents' },
  })
  .addGroup('surface', { ...color, ...border })
  .build();
```

The same collision and overlap rules as `addGroup()` apply:

- Prop names must not match any registered group name.
- Identical prop redefinitions (same property, scale, transform, negative) are tolerated.
- Divergent redefinitions throw.

  <ParamTable
    params={[

        type: 'readonly { toConfig(): SerializedConfig }[]',
        default: 'undefined',
        desc: 'Array of external system instances. Any value from SystemBuilder.build() satisfies the { toConfig(): SerializedConfig } constraint. Read by the extraction pipeline to discover external packages for source walking.',
      },
    ]}
  />
</APIBlock>

<Callout variant="info" title="How includes works">
  `includes` is a static-analysis marker, not a type-level fiction or a runtime
  operation. The extraction pipeline (`discover-packages.ts`) reads the
  constructor-argument AST, traces each identifier back to its import
  declaration, resolves the package specifier, and walks the package's source
  tree for components to extract. Runtime is no-op by design — all work has
  already happened at compile time. This is the authoritative mechanism for
  multi-package design-system composition.
</Callout>

```ts
import { ds as baseDs } from '@acme/design-system';

export const { system: ds } = createSystem({
  includes: [baseDs],
})
  .addGroup('surface', { ...color, ...border })
  .build();
```

## build()

<MethodCard
  name="build"
  description="Finalize the builder — returns the system instance and the global styles factory"
  returnType="{ system: SystemInstance<PropReg, GroupReg>; createGlobalStyles: GlobalStylesFactory }"
/>

<APIBlock>
  <TypeSignature
    name="build"
    params={[]}
    returns="{ system: SystemInstance<PropReg, GroupReg>; createGlobalStyles: GlobalStylesFactory }"
  />
</APIBlock>

Finalizes the builder. Returns two values — always destructure both:

```ts
export const { system: ds, createGlobalStyles } = createSystem()
  .addGroup('surface', { ...color, ...border })
  .build();
```

The `system` return is renamed via destructuring. `ds` is the conventional name. All component authoring calls (`ds.styles()`, `ds.variant()`, etc.) go through this instance.

### SystemInstance

<TypeSignature
  name="SystemInstance"
  generics="PropReg extends Record<string, Prop>, GroupReg extends Record<string, (keyof PropReg)[]>"
  params={[]}
  returns="Animus<PropReg, GroupReg> & { toConfig(): SerializedConfig }"
/>

`SystemInstance` is the built system: an `Animus` builder instance extended with `toConfig()`. The builder chain is available on it. `toConfig()` is called by the Vite plugin subprocess — do not call it in application code.

The generic parameters carry the full accumulated type state, which powers prop type narrowing on every component defined via the system.

### createGlobalStyles

`createGlobalStyles` is a factory for global and reset styles. It accepts a `GlobalStyleMap` — a nested `Record<string, Record<string, any>>` of CSS selector to style object — and returns a `GlobalStyleBlock` that the Vite plugin serializes into `@layer global`.

```ts
export const globalStyles = createGlobalStyles({
  '*, *::before, *::after': { boxSizing: 'border-box' },
  'html, body': { bg: 'bg', color: 'text' },
  '::-webkit-scrollbar': { width: '3px' },
  '::-webkit-scrollbar-thumb': { bg: 'scrollbar' },
});
```

The `GlobalStyleBlock` return type has a `__brand: 'GlobalStyleBlock'` discriminant. The Vite plugin uses this brand to detect global style exports in the system module.
