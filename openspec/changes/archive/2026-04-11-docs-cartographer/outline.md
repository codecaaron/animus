# Animus Documentation Outline

> Generated by docs-cartographer Phase 1. This is the structural contract for Phase 2 (docs-scribe) and Phase 3 (docs-auditor).

## Structural Decisions

### Page Count: 25 (19 repurposed + 6 new)

The seed inventory suggested 13 pages. Full API surface enumeration revealed:
- **13 pre-built prop groups** (not 10) — `mode`, `vars`, `background`, `transitions` were missing from seed
- **27 built-in selectors** with explicit cascade ordering — too dense for a subsection
- **6 built-in transforms** + `createTransform` factory — distinct from system props
- **`declareContextualVars()`**, **`extendScale()`**, **`varRef()`**, **`.manifest`** on ThemeBuilder — not in seed
- **`GlobalStyleBlock.serialize()`** contract — extraction lifecycle hook
- **`SystemBuilder.includes()`** for external system composition — not in seed
- **`asChild`** and **`as`** props on all terminal components — headless composition pattern
- **CompoundEntry conditions accept arrays** — multi-value matching
- **90+ type exports** across 5 categories

### New Pages (6)

| File | Justification |
|------|--------------|
| `architecture/global-styles.mdx` | `createGlobalStyles` factory + `GlobalStyleBlock.serialize()` contract + reset/global layers + keyframe patterns. 6+ APIs, distinct lifecycle from component styling. |
| `authoring/selectors.mdx` | 27 built-in selectors + `addSelectors()` + custom selectors + cascade ordering + `.states()` integration. Would overwhelm "Variants & States" if merged. |
| `authoring/custom-props.mdx` | `.props()` + `CustomPropConfig` + `createTransform` + 6 built-in transforms + scale binding + variable declaration. Distinct concept from system prop groups. |
| `advanced/extraction.mdx` | @layer structure, dev vs prod, stable class names, per-property skip, virtual modules, HMR. Conceptual explanation critical for debugging and architecture understanding. |
| `authoring/recipes.mdx` | Full component builds combining 3-5 concepts (variants + states + selectors + system props + composition). Three reviewers independently flagged the gap between concept pages and real-world usage. |
| `support/migration.mdx` | Adoption guide for teams coming from runtime CSS-in-JS (styled-components, Emotion, Chakra). Covers coexistence, incremental migration, common refactoring patterns. All three reviewers flagged this gap. |

### Recommended Reading Order

Three-lens review identified that architecture-first ordering (7 setup pages before authoring) creates friction for new users. The recommended reading order for the sidebar navigation is:

1. Introduction → Getting Started (onboarding)
2. Base Styling → Variants & States → Selectors → System Props → Custom Props → Composition → Recipes (authoring — learn to USE it)
3. Theming → Color Modes → System Setup → Theme Extension → Global Styles (architecture — learn to CONFIGURE it)
4. Vite Plugin → Next.js & Remix (integration)
5. TypeScript → Framework Agnostic → Extraction & CSS Output (advanced)
6. Reference pages (builder chain, createTheme, createSystem, compose)
7. Troubleshooting → Migration (support)

Phase 2 should update `docsNav.ts` to reflect this order.

### The Static Constraint

**The single most important constraint**: all style values must be **static string literals** for extraction. No template literals, no function calls, no variables. This was independently flagged by all three reviewers as insufficiently surfaced. Phase 2 must:
- State this constraint prominently on the Introduction page (Section 2, "What Animus Does Differently")
- Include it in Getting Started as a first-class callout
- Repeat a standard callout on EVERY authoring page (8-12) linking to the Per-Property Skip Model explanation
- The Troubleshooting page should have "partial extraction" as a primary diagnostic entry

### Merged Pages

| Original Suggestion | Merged Into | Reason |
|--------------------|------------|--------|
| "Prop shorthand and custom props" | Split: shorthand in System Props, custom in Custom Props & Transforms | Different audiences: shorthand is consumption, custom is authorship |
| "Theme distribution" | Theme Extension page | `.from()` + `.extend()` + module augmentation fits naturally here |
| "API reference catch-all" | Distributed across Reference section | Builder Chain, createTheme, createSystem, compose each have enough depth |

   - Intent: Constraint surfacing — the #1 thing every developer must know before writing code
   - APIs covered: None (constraint, not API)
   - Components: `<Callout variant="warn">`, `<BeforeAfter>` (what works vs what gets skipped)

4. **The Zero-Runtime Proof**
   - Intent: Trust builder — show exactly what ships to the browser
   - APIs covered: `system#createComponent`, `system#createClassResolver` (as extraction output, not consumer API)
   - Components: `<BundleBar>`, `<MetricCard>`, `<CodeExample>` (builder chain → extracted CSS + transformed JS → bundle)
   - Edge cases: `createComponent` IS runtime code — it resolves class names from props at render time. Be explicit about this residual: it's a thin prop-to-className mapper, not style computation. Show the actual bundle contribution.
   - Cross-refs: → Framework Agnostic (runtime API details)

5. **Who This Is For**
   - Intent: Audience fit — design system authors, component library maintainers, teams with mature token systems

   - Components: `<Callout variant="tip">`

### Page 9: Variants, Compounds & States

- **File:** `content/authoring/variants-states.mdx`
- **URL:** `/docs/authoring/variants-states`
- **Diataxis:** How-To
- **Reader context:** Developer who can write base styles and wants conditional styling. Understands the concept of variants (size, color) from other systems.
- **Objective:** Reader can define named variants, compound conditions, and boolean state toggles, and understands the three conditional styling tools and when to use each.

#### Sections

1. **Three Tools for Conditional Styles**
   - Intent: Decision guide FIRST — map before territory
   - APIs covered: (overview, not detailed API)
   - Components: `<TabGroup>` (side-by-side comparison), `<Callout variant="tip">`, `<ParamTable>`
   - Edge cases: Variants = enumerable named options (size: sm/md/lg) — consumer picks from a list. States = boolean toggles for UI state (isLoading, isOpen, copied) — component controls internally. Compounds = multi-variant cross-conditions (size=lg AND color=danger). The same visual effect CAN be achieved with either — choose based on semantic intent. **States are NOT CSS pseudo-class sugar** — they're boolean props, not selectors. Selector aliases (`_hover`, `_focus`) are a separate orthogonal concept (see Selectors & Nesting).
   - Cross-refs: → Selectors & Nesting (selector aliases are NOT states), → Composition (shared variants)

2. **Named Variants**
   - Intent: API reference — .variant() config shape
   - APIs covered: `system#AnimusWithBase.variant`, `system#VariantConfig`
   - Components: `<MethodCard>`, `<ParamTable>`, `<SyntaxBlock>`, `<LivePreview>`
   - Edge cases: `prop` key overrides the prop name (defaults to a generated name). `defaultVariant` sets the fallback when no prop is provided. `base` styles apply to ALL variant options (merged before specific variant styles). Variants can be chained — multiple `.variant()` calls. Empty variant values (`{}`) are valid for semantic grouping without visual change.
   - Cross-refs: → Builder Chain (type-state ordering)
