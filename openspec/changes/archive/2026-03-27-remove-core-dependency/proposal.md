## Why

The vite-plugin still references `@animus-ui/core` in two places:

1. **`loadConfig()` fallback** — when no `system` option is provided, the plugin falls back to `require('@animus-ui/core').getExtractConfig()` to load the prop registry and group map.
2. **`resolve-transforms.ts`** — a standalone subprocess script that imports all 12 prop groups directly from core to build a transform function registry.

Both paths are vestigial. The `system` option (introduced with the system package) provides everything via `serialize()`: `propConfig`, `groupRegistry`, `transforms`, and `globalStyles`. Core is not in the plugin's `package.json` dependencies — it's loaded at runtime via subprocess, creating an implicit, undeclared dependency.

The transform registry is the key issue: transforms are NOT a separate concern. They are carried by the groups themselves. When you call `.addGroup(space)`, that group's props carry their transform functions. `serialize()` extracts them. The core import was a shortcut to get the same data from a static, pre-built registry — but it's the wrong source because it doesn't reflect the user's actual configuration.

Additionally, the in-process `transformRegistry` Map declared at line 569 is **never populated** (no `.set()` calls exist). It's dead code that serves only as a broken fallback.

## What Changes

### vite-plugin/src/index.ts

1. **Remove `loadConfig()` function entirely** — `system` is the only supported config path.
2. **Remove `configPath`, `config`, and `groupRegistry` options** from `AnimusExtractOptions`. The `system` option replaces all three.
3. **Remove the `transformRegistry` Map** — never populated, dead weight.
4. **Remove `resolveTransformPlaceholders()` function** — it calls `resolve-transforms.ts` which imports core. The system resolve script (already generated by `loadSystem()`) handles this.
5. **Simplify transform resolution in `runAnalysis()`** — always use the system resolve script. No fallback to the legacy path.
6. **Update option JSDoc comments** — remove references to core auto-import.

### vite-plugin/src/resolve-transforms.ts

**Delete entirely.** Its only purpose is loading transforms from core. The system resolve script generated at `loadSystem()` lines 872-897 already does this correctly from `system.serialize().transforms`.

### extract/src/ (Rust)

**No changes needed.** The Rust crate:
- Already recognizes both `@animus-ui/core` AND `@animus-ui/system` as valid import sources for builder chains (transform_emitter.rs line 293).
- Test fixtures that import from `@animus-ui/core` are valid — core is a consumer package that re-exports system. Consumers can still use it.

## Impact

- **`packages/vite-plugin/src/index.ts`** — remove loadConfig, configPath/config/groupRegistry options, transformRegistry Map, resolveTransformPlaceholders
- **`packages/vite-plugin/src/resolve-transforms.ts`** — delete
- **`packages/vite-plugin/dist/resolve-transforms.d.ts`** — will disappear on next build

No changes to runtime, system, extract, core, or any consumer-facing API. Core continues to exist as a consumer convenience package — it's just no longer referenced by the build pipeline.

## Migration

The `system` option is already the documented and recommended path. Anyone using the legacy `configPath` option would need to switch to:

```typescript
animusExtract({ system: './src/system.ts' })
```

This was already the only working path for theme-manifest, global styles, custom props, and all features shipped in sessions 11-16.
