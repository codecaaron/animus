## Context

The showcase docs use a two-column layout: a sticky left sidebar listing all doc pages, and a content area. Long pages (Core Concepts, API Reference) have all topics on a single page. A table of contents was recently added to the sidebar below the page links, but mixing page-level navigation and within-page anchors in one column creates visual noise and scaling problems.

The current routing structure is flat: `/docs`, `/docs/start`, `/docs/concepts`, `/docs/api`, `/docs/examples`. Each route maps to a single monolithic page file. The sidebar component has a hardcoded link list.

Components are built with the `ds.styles()` Animus builder chain and extracted at build time. All style values must be static string literals. The app uses React Router v6 with nested `<Route>` elements and layout routes via `<Outlet>`.

## Goals / Non-Goals

**Goals:**
- Separate section identity, page selection, and within-page orientation into three distinct UI regions
- Enable individual topic pages that can be linked directly (e.g., `/docs/concepts/design-tokens`)
- Provide a right-column ToC with scroll spy for within-page navigation
- Maintain the terminal-forward aesthetic — monospace, minimal, no heavy borders
- Support responsive collapse: right ToC first, then left sidebar
- Keep content column wide enough for code blocks at common viewports

**Non-Goals:**
- Search functionality
- Mobile-first hamburger/drawer navigation (desktop docs experience is primary)
- Breadcrumbs or other tertiary navigation elements
- Automated heading extraction for sidebar (sidebar structure is route-driven, not DOM-driven)
- Changing doc content — only restructuring files and routing

## Decisions

### 1. Route structure: nested routes per section

Each doc section that has multiple topics gets nested routes:

```
/docs                    → Why Animus (index)
/docs/start              → Getting Started (flat)
/docs/concepts           → redirect to /docs/concepts/builder-chain
/docs/concepts/:topic    → individual concept pages
/docs/api                → redirect to /docs/api/create-theme
/docs/api/:topic         → individual API pages
/docs/examples           → Examples (flat)
```

**Why nested over flat:** Flat routes (`/docs/builder-chain`) lose the section context from the URL. Nested routes make the hierarchy explicit and allow section-level layout wrappers.

**Alternative considered:** Keep monolithic pages, use hash-based navigation only. Rejected because hash navigation doesn't trigger route-based code splitting, doesn't support section-level redirects, and the sidebar hierarchy would be misleading (looks like route links but behaves as scroll targets).

### 2. Sidebar driven by route config, not DOM

The sidebar renders from a static route configuration object, not from querying the DOM or reading file system. Each section declares its label, base path, and child pages.

```typescript
const DOCS_NAV = [
  { label: 'Why Animus', path: '/docs', children: [] },
  { label: 'Getting Started', path: '/docs/start', children: [] },
  {
    label: 'Core Concepts',
    path: '/docs/concepts',
    children: [
      { label: 'The Builder Chain', path: '/docs/concepts/builder-chain' },
      { label: 'The Cascade Contract', path: '/docs/concepts/cascade-contract' },
      // ...
    ],
  },
  // ...
];
```

**Why static config:** Route configuration is the source of truth for what pages exist. The sidebar should reflect routes, not headings. This avoids DOM timing issues, works with SSR/SSG if needed later, and makes the navigation predictable.

### 3. Right-column ToC via heading scan + IntersectionObserver

The right-column ToC scans `h2[id]` and `h3[id]` elements on mount and route change. Active heading is tracked via IntersectionObserver with `rootMargin: '0px 0px -70% 0px'` to trigger activation when a heading enters the top 30% of the viewport.

**Why DOM scan:** Heading IDs are already auto-generated by the `Heading` component. Scanning the DOM is simpler than requiring each page to export a heading manifest. The ToC is a read-only view of what's on the page.

**Why not in the sidebar:** Mixing anchors and routes in one column was the problem this change solves. The right column is dedicated to within-page orientation.

### 4. Three-column layout with CSS grid

Replace the current flexbox two-column layout with CSS grid:

```
grid-template-columns: 200px 1fr 180px
```

At `md` breakpoint and below, collapse to two columns by hiding the right ToC. The sidebar persists at all breakpoints where horizontal space permits.

**Why grid over flex:** Grid makes the three-column layout explicit and allows the right column to collapse without affecting the other two columns. The `1fr` center column naturally absorbs available space.

### 5. Page splitting strategy

Only pages with 5+ major sections get split:
- **Core Concepts** → 6 pages (builder-chain, cascade-contract, design-tokens, responsive-props, variants-states, slot-composition)
- **API Reference** → 6 pages (create-theme, create-system, builder-chain, create-transform, prop-groups, vite-plugin)
- **Why Animus** → stays as single page (5 sections, narrative flow benefits from being one page)
- **Getting Started** → stays as single page (sequential steps)
- **Examples** → stays as single page (interactive demos)

**Why not split everything:** Short pages with sequential flow (Getting Started) or narrative arc (Why Animus) read better as single pages. The split threshold is "would a user want to link directly to this section independently?"

### 6. Heading scroll offset

Add `scroll-margin-top` to heading elements in the `Heading` component to account for the sticky top nav (~48px). This ensures anchor links don't scroll the heading behind the nav bar.

```typescript
// In Heading base styles
scrollMarginTop: '64px', // nav height + breathing room
```

## Risks / Trade-offs

- **[Breaking links]** → Existing URLs like `/docs/concepts` change meaning (becomes redirect instead of full page). Mitigation: section index routes redirect to the first child page, so old links still work — they just land on a subset of content.
- **[Content duplication risk]** → Splitting pages means shared context (imports, intro text) could be duplicated. Mitigation: keep each page self-contained. Shared components are already in the component library.
- **[Right ToC on short pages]** → Pages with 0-1 headings waste the right column. Mitigation: ToC component auto-hides when fewer than 2 headings are found. Layout falls back to two-column.
- **[Viewport squeeze]** → Three columns at narrow desktop viewports could compress content. Mitigation: right ToC collapses at 1024px. At 1280px, content gets 200 + 32 + content + 32 + 180 = content gets ~836px minimum, well above the 700px target.
