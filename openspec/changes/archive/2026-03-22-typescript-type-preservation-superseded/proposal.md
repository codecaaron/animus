## Original Premise (Incorrect)

This proposal assumed that `createComponent` needed generic type parameters to preserve TypeScript type safety after extraction. The concern was that `<Button variant="typo">` would no longer be a type error after the builder chain was replaced.

## Why This Is Unnecessary

The transform output is an ephemeral artifact — generated by Vite's build pipeline, consumed by esbuild (which strips types), and served to the browser as plain JS. **Nobody type-checks the transform output.**

Type safety lives at the SOURCE level:

1. Developer writes `components.tsx` using the builder chain → tsc checks it → type-state machine catches invalid chains and props
2. Developer writes `App.tsx` using `<Button variant="ghost">` → tsc checks it → import resolves to source types
3. Vite transforms → replaces chain with `createComponent(...)` → esbuild strips types → browser gets JS
4. The replacement happens AFTER type checking. The erased types were never needed post-transform.

## What Was Done Instead

The `extraction-correctness` change addresses the actual concerns:
- **Source-level type verification**: Smoke test tsconfig + `tsc --noEmit` (validates that source types actually work)
- **Rust defense-in-depth**: Unknown chain method bail (the build-time mirror of the type system)
- **Dead import removal**: Transform strips `@animus-ui/core` import so Emotion doesn't load
- **`.asComponent()` coverage**: Exercises the HOC className forwarding path

## Recommendation

Archive this change. The generic `createComponent<Props>` approach is unnecessary engineering for a problem that doesn't exist in the Vite pipeline.
