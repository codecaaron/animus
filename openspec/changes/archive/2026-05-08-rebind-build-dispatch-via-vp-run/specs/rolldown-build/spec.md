## MODIFIED Requirements

### Requirement: Rolldown as library bundler

Each published TypeScript package in the active build graph (`packages/extract`, `packages/system`, `packages/properties`, `packages/vite-plugin`, `packages/next-plugin`, and the private workspace `packages/_assertions`) SHALL use Rolldown as the bundling engine to produce ES module bundles. The orchestrator binding (currently `tsdown`, which wraps Rolldown internally) SHALL handle TypeScript and JSX compilation natively without Babel; declaration emission SHALL be handled by `tsgo` (TypeScript 7 native preview) per the canonical `build:ts` per-package invocation (`tsdown && tsgo -p tsconfig.build.json`).

#### Scenario: System package build

- **WHEN** `vp run build:ts` executes (which dispatches to `bun run --filter './packages/*' build:ts`)
- **THEN** Rolldown (via tsdown) produces ES module output for each package's configured entries
- **AND** all `node_modules` are externalized from the bundle
- **AND** the output extension follows the package's `type` field (`.js` for `type: "module"` packages, the canonical animus configuration)

#### Scenario: TypeScript declaration output

- **WHEN** a published package's `build:ts` script completes
- **THEN** `dist/<entry>.d.ts` declaration files are generated by `tsgo -p tsconfig.build.json`
- **AND** the declaration extension is `.d.ts` (not `.d.mts`), preserving the consumer-visible `package.json exports.types` contract

### Requirement: Shared Rolldown base config

A shared bundler base configuration SHALL be defined at the repository root for the Rolldown-based build pipeline. Per-package configs SHALL extend the shared base with minimal overrides (entry points, externals). The current shared base is `tsdown.config.base.ts` (tsdown wraps Rolldown internally), consumed by per-package `tsdown.config.ts` files via the `createConfig()` helper. Under engine-identity-only interpretation per Requirement "Binding to orchestration-architecture", the shared-config surface MAY be replaced by an alternative wrapper's equivalent (e.g., `vite.config.ts pack:` blocks for `vp pack`, or `rolldown.config.ts` for direct Rolldown invocation) in a future engine-swap slice provided the output is functionally equivalent.

#### Scenario: Per-package config

- **WHEN** examining a package's `tsdown.config.ts`
- **THEN** it imports and extends the root shared `tsdown.config.base.ts` via the `createConfig()` helper
- **THEN** package-specific config is no more than ~10 lines (typically just an `entry: [...]` array; format/dts/clean/outDir/target/platform inherited from the shared base)

### Requirement: Binding to orchestration-architecture

The library bundler contract — Rolldown as the bundling engine, ES-module output, TypeScript and JSX compilation without Babel, externalization of `node_modules`, declaration emission — defined in this spec SHALL be realized through the orchestrator binding designated by the `orchestration-architecture` capability. The current binding invokes `tsdown` (which uses Rolldown internally) per package via the orchestrator's task graph: `vp run build:ts` (defined in `vite.config.ts run.tasks`) dispatches `bun run --filter './packages/*' build:ts`, and each per-package `build:ts` script invokes `tsdown && tsgo -p tsconfig.build.json` (tsdown emits ES module bundles; tsgo emits `.d.ts` declarations).

The `orchestration-architecture` capability's "Follow-On Policy Decomposition" requirement enumerating "A change rebinding the library bundler" is satisfied by THIS capability's binding through `vp run build:ts`. The bundler-engine identity (Rolldown), the output format (ES module), the externalization contract, and the declaration-emission requirement are preserved. Under engine-identity-only interpretation, the wrapper (currently `tsdown`) is negotiable as long as engine identity is preserved; the wrapper choice is an orchestrator-binding concern, not a spec-level invariant of this capability.

A future engine-swap policy slice — `migrate-build-to-vp-pack` — is RESERVED for replacing the wrapper from `tsdown` to `vp pack` when vp pack reaches GA and accumulates sufficient bake time. That future slice SHALL preserve the bundler-engine identity (Rolldown), the output format (ES module), the externalization contract, and the declaration-emission requirement. The future slice MAY replace `tsdown.config.ts` files with `vp pack` configuration (CLI args, per-package `vite.config.ts pack:` blocks, or the orchestrator's equivalent config surface) provided the output is functionally equivalent and the consumer-visible package contract (`.js` + `.d.ts` exports paths in `package.json exports`) is either preserved or migrated explicitly with consumer-side coordination.

#### Scenario: Rolldown engine identity is preserved through current binding

- **WHEN** `vp run build:ts` executes against any published package in the active build graph
- **THEN** the underlying bundling engine is Rolldown (via tsdown wrapper)
- **AND** ES-module output is emitted
- **AND** all `node_modules` are externalized from the bundle
- **AND** TypeScript declaration files are emitted (via tsgo)

#### Scenario: Future engine-swap preserves engine identity

- **WHEN** the future `migrate-build-to-vp-pack` cutover slice rebinds the library-build invocation from `tsdown` to `vp pack`
- **THEN** the underlying bundling engine continues to be Rolldown (vp pack wraps Rolldown internally)
- **AND** ES-module output continues to be the only emitted format
- **AND** all `node_modules` continue to be externalized from the bundle
- **AND** TypeScript declaration files continue to be emitted
- **AND** the consumer-visible package contract (`.js` + `.d.ts` exports paths) is preserved or migrated with consumer-side coordination
